# AI车辆数据实时监测平台

这是一个面向预筛选作业的可运行 Web 产品原型。原型聚焦车队运营和电池健康分析团队在真实工作中的核心场景：从车队中发现异常车辆，按时间段查看单车轨迹、遥测、事件和 BMS 变化，并用透明的 AI 式研判帮助决定下一步排查动作。

## 运行方式

```bash
npm install
npm run dev
```

启动后访问终端显示的本地地址。构建验证命令：

```bash
npm run build
```

## 推荐验证路径

1. 打开页面后保持默认时间窗 `10:00-11:30 异常聚焦`。
2. 选择 `苏A-D9187`，查看电池温升、SOC 下降、BMS 评分和事件时间线。
3. 阅读右侧 `AI辅助研判`，确认每条建议都能追溯到具体证据。
4. 切换到 `苏A-P3275`，观察数据上报中断场景。
5. 在筛选区按状态、风险、分组或关键词过滤车辆，验证列表、汇总和选中车辆上下文同步变化。

## 产品判断

目标用户：

- 车队运营分析员：需要快速识别哪些车辆影响任务履约。
- 平台支持工程师：需要定位上报缺口、轨迹异常和设备通信问题。
- 电池健康分析员：需要观察 SOC、温度、绝缘、总压和 BMS 评分之间的关系。

产品定位：它不是原始日志浏览器，而是一个时段化的车辆研判工作台。核心状态是“选中的车辆 + 选中的时间窗”，所有轨迹、指标、事件和 AI 研判都围绕这个状态同步刷新。

优先解决的问题：

- 从车队层面快速找到风险车辆。
- 按具体时间段查看单车详细数据。
- 把轨迹、驾驶、电池、BMS 和事件放在同一个判断上下文里。
- 让 AI 建议可解释、可追溯，而不是只给结论。

暂不解决的问题：

- 真实设备接入、消息队列、流式计算和持久化存储。
- 权限、组织架构、审计、账号体系。
- 真实地图 SDK 和真实 LLM API。
- 全量车辆信号和完整 BMS 协议覆盖。

## 交互判断

- 使用左侧车队筛选和队列，保证用户先完成“找车”。
- 使用顶部时间窗控制，让所有分析面板统一响应同一个时段。
- 使用轨迹、同步指标、事件时间线和 AI 研判并列呈现，减少来回切页面的成本。
- 对无匹配车辆、无轨迹、指标不足和低风险场景都给出明确状态，避免误导。

## 工程判断

技术栈：Vite + React + TypeScript + 本地确定性数据。

这样取舍的原因：

- 原型重点是产品、交互和判断闭环，前端优先能让审阅方最快运行和验证。
- TypeScript 明确了车辆、遥测、事件、诊断等数据结构。
- 本地数据保证每次打开都能看到相同异常场景，便于审阅和复现。
- AI 辅助研判采用规则模拟，输出风险等级、证据、可能原因和下一步动作，避免真实模型 API 带来的密钥、延迟和不确定性。

如果演进到生产版本，建议补充：

- 车辆遥测接入层：MQTT/Kafka/HTTP ingestion。
- 时间序列存储：按车辆、信号、时间窗口建立索引。
- 实时计算：在线状态、采样缺口、热风险和 BMS 异常的流式规则。
- AI 服务：把规则证据、历史维修记录、任务信息和专家知识输入模型，生成可追溯建议。
- 权限和审计：按车队、角色、数据范围控制访问。

## AI 使用说明

本作业中使用 AI 协助完成了以下环节：

- 阅读和拆解题目需求，形成产品定位、用户场景和功能优先级。
- 生成 OpenSpec 提案、设计、规格和任务拆分。
- 辅助实现 React/TypeScript 原型、模拟数据和诊断规则。
- 辅助检查交付文档是否覆盖运行、验证、产品判断、工程边界和 AI 使用说明。

AI 建议的采纳标准：

- 是否直接服务题目目标，而不是堆功能。
- 是否能被可运行原型验证。
- 是否让交互更清晰，或让工程边界更容易解释。
- 是否保持可复现和可审阅，避免把不可验证的模型能力包装成已实现能力。

页面中的 `AI辅助研判` 是原型内的规则化模拟，不是外部大模型实时生成。这样做是为了让审阅者看到 AI 产品形态、解释结构和决策辅助方式，同时保证演示稳定。
